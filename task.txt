1. Assignment Overview & Vision
You are required to design and build a full-stack, AI-powered smart dining assistant for a restaurant/café
environment. This is not merely a menu-ordering app — it is an AI-first product where every customer touchpoint
is augmented by intelligent agents that understand context, personalise recommendations, proactively upsell,
and collaborate in real time to serve multiple diners simultaneously.
▶ What makes this "AI-First"?
· AI is the primary interaction layer, not an add-on chatbot widget.
· Every recommendation, filter, and upsell is generated by a dedicated AI agent.
· Agents share memory and context — the system learns within a session.
· The UI adapts based on AI signals (e.g., popular right now, trending at lunch).
· Natural language is the default input method; menus are a fallback, not the primary.

▶ Core User Journey (one sentence)
Customer scans QR → lands on menu → AI greets them by mood → conversation drives the order → group
syncs in real time → checkout with OTP → kitchen notified → bill settled.

2. Objectives & Business Goals
2.1 Customer-Facing Goals
· Reduce average decision time from ~8 min to < 2 min using AI guidance.
· Enable fully natural-language ordering in English, Hinglish, and Telugu-English.
· Support group ordering without confusion or cart conflicts.
· Provide a conversational, warm, and witty dining experience.
2.2 Business Goals
· Increase average order value (AOV) via intelligent upselling and bundling.
· Surface high-margin items through contextual placement, not hard pushes.
· Capture lightweight preference data to improve future recommendations.
· Reduce staff dependency for order-taking.
2.3 Technical Goals
· Build a production-grade, demonstrable system within 48 hours.
· Implement a multi-agent AI layer with clear separation of concerns.
· Achieve < 2 s page load and < 3 s AI response latency.
· Design for horizontal scalability from day one.

3. AI-First Philosophy & Agent Architecture
Traditional ordering apps treat AI as a chatbot sidebar. In this system, AI agents are first-class citizens that own
specific responsibilities and coordinate through a central orchestrator. Each agent has tools, memory access, and
a defined persona.
3.1 Agent Roster
· Greeter Agent:
Welcomes users, identifies mood/preference via a 2-question micro-onboarding, sets session
context.
· Recommendation Agent:
Core menu intelligence. Matches user intent to menu items using semantic search + LLM
reasoning.
· Upsell Agent:
Monitors cart composition and triggers contextual add-on / combo suggestions at the right
moment.
· Context Memory Agent:
Maintains conversation history, preference state, and cart snapshot within the session.
· Group Coordinator Agent:
Merges individual user intents within the same tableId, detects conflicts, suggests shared items.

· Sentiment & Feedback Agent:
Detects frustration or confusion in messages; escalates or rephrases AI responses accordingly.
· Multilingual NLU Agent:
Normalises mixed-language inputs (Hinglish, Telugu-English) into structured intent before
passing to other agents.
· Order Validation Agent:
Final check before order submission: confirms items in stock, validates quantities, applies
business rules.
3.2 Agent Interaction Flow
User Input
│
▼

│
▼

├─► Greeter Agent (first message only)

3.3 Agent Tools (Function Calling)
Each agent has access to a defined set of tools via the LLM function-calling API:
search_menu(query, filters) → Vector/semantic search over menu items
get_cart(tableId) → Fetch current cart state for a table
add_to_cart(tableId, itemId, qty) → Add item to shared cart
get_popular_items(time_of_day) → Return top-ordered items for current hour
get_complementary(itemId) → Return items frequently ordered together
get_session_context(tableId) → Return full conversation + preference history
update_preference(tableId, key, val) → Persist a user preference
validate_stock(itemId) → Check real-time availability
send_otp(phone) → Trigger OTP verification
create_order(tableId) → Finalise and submit order to kitchen

→ Allergen filter applied; spicy + no-dairy items surfaced.

5.3 AI Tone & Persona Guidelines
· Name: "Zara" (configurable per restaurant brand)
· Tone: Warm, energetic, slightly witty — like a knowledgeable friend at the table.
· Never robotic; avoid "I am an AI" disclaimers in the ordering flow.
· Use humour sparingly; always prioritise accuracy over cleverness.
· In multilingual inputs, respond in the same language/mix the user used.
· Keep responses concise: max 3 items per suggestion, max 2 sentences of copy.
5.4 Upselling Logic (Mandatory)
The Upsell Agent fires at these trigger points:
Trigger: After add-to-cart
Copy: "Great choice! Most people pair {item} with {complement}. Want to add it?"
Trigger: Cart total crosses ₹500
Copy: "You're ₹{X} away from our Meal Deal — add {item} to unlock it."
Trigger: Cart has mains, no beverage
Copy: "Looks like you're missing drinks! Want something refreshing?"
Trigger: Cart has veg items only
Copy: "Feeling adventurous? Our {non-veg item} is today's chef special."
Trigger: Time of day = evening
Copy: "Evening special: {dessert} is half-price until 8 PM."
Trigger: User says "that's all"
Copy: "Before you go — {high-margin item} takes only 5 mins and pairs perfectly with what you
have."

5.6 Multilingual Handling
The Multilingual NLU Agent must handle:
· Pure English inputs
· Hinglish: "mujhe ek spicy starter aur ek light main chahiye"
· Telugu-English: "konchem spicy ga undali, veg kaadu"
· Typos & slang: "sumthing swt plz"
Implementation: pre-processing step calls GPT with a normalisation prompt that outputs structured JSON
{intent, preferences, language_detected, raw_text} before routing to other agents.
5.7 Contextual Intelligence

· Time-of-day awareness: Breakfast (7–11), Lunch (12–15), Evening (16–19), Dinner (19–23)
· Current cart awareness: never suggest items already in cart
· Session preference memory: if user said "no dairy", honour it throughout
· Group size inference: detect plural pronouns ("we", "our") to adjust portions/quantities suggested
· Popularity signals: surface items with high order velocity in last 2 hours

6. Multi-Agent Orchestration Design
6.1 Orchestration Pattern
Use a Router-Orchestrator pattern: a lightweight LLM call classifies the user message into an intent category,
then dispatches to the appropriate specialist agent. This avoids a single monolithic prompt and keeps latency low.
Intent Classification (fast, gpt-4o-mini)
├─ GREET → Greeter Agent
├─ RECOMMEND → Recommendation Agent + Context Memory Agent
├─ ADD_ITEM → Recommendation Agent (confirm) + add_to_cart tool
├─ UPSELL_CHECK → Upsell Agent (async, non-blocking)
├─ GROUP_MERGE → Group Coordinator Agent
├─ CHECKOUT → Order Validation Agent
├─ MULTILANG → Multilingual NLU Agent (pre-step, always)
└─ FALLBACK → General LLM with menu context injection
6.2 Memory Architecture
Three tiers of memory per session:
· Working Memory (in-request):
Current message + last 5 exchanges. Injected into every agent prompt.
· Session Memory (Redis):
Full conversation summary, preferences, cart snapshot. TTL = session duration.
· Long-Term Memory (optional):
Cross-session preferences if user provides phone. Enables personalisation on return visits.

6.3 Prompt Engineering Standards
· System prompt includes: restaurant name, menu summary, current time, table context, user
preferences.
· Use few-shot examples in Recommendation Agent prompt for consistent output format.
· All agent prompts follow ROLE → CONTEXT → TASK → FORMAT → CONSTRAINTS structure.
· Output schema for recommendation responses: JSON array of {itemId, name, reason, price}.
· Temperature: 0.7 for conversational agents, 0.2 for validation/structured agents.
· Max tokens capped per agent to control cost: Greeter=150, Recommendation=400, Upsell=200.
6.4 Agentic RAG for Menu Search
Menu items are embedded at startup using text-embedding-3-small. The Recommendation Agent performs a
cosine-similarity search against the vector store before calling the LLM, reducing hallucination and grounding
responses in actual available items.
User: "something light and tangy"
1. Embed query → vector
2. cosine_search(vector, top_k=10)
3. Inject top-10 item summaries into LLM prompt

4. LLM selects best 3 with reasons
5. Return structured {itemId, name, reason, price}[]

12. Security & Compliance
12.1 Input Validation & Sanitisation

· All user inputs validated via Zod (TypeScript) or Joi (JavaScript) before processing.
· AI chat inputs: max 500 chars, strip HTML/JS injection attempts before sending to LLM.
· Phone numbers validated with libphonenumber-js.
· OTP: HMAC-signed, 5-minute TTL, max 3 attempts before lockout.
12.2 API Security
· Rate limiting: express-rate-limit; 60 req/min per IP; 20 AI requests/min per session.
· CORS restricted to known frontend origin.
· sessionId stored as HttpOnly cookie; not exposed to JavaScript.
· No PII stored in AI prompts; phone numbers hashed in logs.
12.3 LLM Security
· Prompt injection defence: user input is always in a sandboxed <user_message> tag; system instructions
are not overridable.
· Output filtering: AI responses scanned for PII patterns before sending to client.
· Model responses validated against JSON schema before rendering.
· No tool execution without explicit intent classification; agents cannot call privileged tools without routing
through orchestrator.
12.4 Data Privacy
· Phone numbers and names collected only at checkout; not stored in AI context.
· Session data purged after order completion + 24 hours.
· GDPR/PDPA compliant: minimal data collection, clear purpose.
